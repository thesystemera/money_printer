Clean three-tier options analytics system that processes current snapshots and historical data

CORE ARCHITECTURE PRINCIPLES:
-----------------------------
1. CLEAN PIPELINE: Collect Raw Data → Transform to UnifiedContract Objects → Calculate Baselines → Process → Analyze
2. HISTORICAL INTELLIGENCE: All thresholds derived from actual market patterns, not constants
3. DATA AWARENESS: System adapts to available data types (premium/volume/OI/Greeks/IV)
4. NO BUCKET PRE-PROCESSING: Raw data stays raw, bucket analysis happens on-demand in processing layers
5. ADAPTER PATTERN: Handle complexity of different API response formats cleanly
6. THREE-TIER INSIGHTS: Current + Historical + Comparative Intelligence

API DATA SOURCES & MULTIPLE ENDPOINTS:
======================================

POLYGON API ENDPOINTS USED:
---------------------------

1. CURRENT SNAPSHOT ENDPOINT:
   📍 GET /v3/snapshot/options/{symbol}
   ✅ Returns: Contract volume, open interest, Greeks, implied volatility, bid/ask
   🎯 Used for: Real-time current analysis
   📊 Data Quality: High - single API call gets all current data

2. HISTORICAL CONTRACTS ENDPOINT:
   📍 GET /v3/reference/options/contracts
   ✅ Returns: Contract definitions (strikes, expirations, tickers) for specific date
   🎯 Used for: Discovering what contracts existed historically
   📊 Data Quality: Complete contract universe for date

3. HISTORICAL VOLUME ENDPOINT (Multiple Calls):
   📍 GET /v2/aggs/ticker/{option_ticker}/range/1/minute/{date}/{date}
   ✅ Returns: Minute-level volume and pricing data for specific contract
   🎯 Used for: Historical volume/premium analysis per contract
   📊 Data Quality: Granular but requires separate call per contract (sampling used)

WHY MULTIPLE API CALLS ARE NECESSARY:
------------------------------------
Historical analysis requires BOTH contract definitions AND volume data:

STEP 1: Get Contract Universe
- API Call: /v3/reference/options/contracts?as_of={date}
- Returns: ["AAPL_150C_2024-01-19", "AAPL_155C_2024-01-19", ...]
- Purpose: Know what strikes/expirations existed on historical date

STEP 2: Get Volume Data for Each Contract (Sampled)
- API Calls: /v2/aggs/ticker/{each_contract}/range/1/minute/{date}/{date}
- Returns: Minute-level volume/price data per contract
- Purpose: Calculate historical P/C ratios, premiums, activity levels
- Optimization: Sample 20% of contracts to reduce API calls (configurable)

WHY ADAPTERS ARE ESSENTIAL:
--------------------------
Different endpoints return fundamentally different data structures:

Tier1Adapter (Current Snapshot):
- Input: Rich snapshot with nested structure (details, day, greeks, last_quote)
- Transforms: Single API response → Multiple UnifiedContract objects
- Handles: Bid/ask calculations, Greeks extraction, volume normalization

Tier2Adapter (Historical Data):
- Input: Contract definitions + Volume data from multiple endpoints
- Transforms: Combined data from 2+ API calls → UnifiedContract objects
- Handles: Ticker parsing, volume aggregation, premium calculations

API RATE LIMITING & OPTIMIZATION:
=================================

CURRENT SNAPSHOT: 1 API call
- Single endpoint provides all current data
- Cached for 1 hour
- No rate limiting concerns

HISTORICAL DATA: 2 + N API calls per date
- 1 call: Get contract definitions
- 1 call: Sample contracts (20% of total)
- N calls: Get volume data for sampled contracts
- Cached for 6 months per date
- Rate limited with semaphore (50 concurrent)
- Intelligent sampling reduces API usage by 80%

STOCK DATA: 1 API call via stock service
- Alpaca integration for minute-level stock data
- Used for price embedding and volatility calculations

========================

- Tier 1: bid/ask midpoint when available → premium = volume × price × 100
- Tier 2: volume × avg_price × 100 as premium proxy

SENTIMENT ANALYSIS HIERARCHY:
1. Premium-based P/C ratios (most accurate)
2. Volume-based P/C ratios (good proxy)
3. Open Interest ratios (baseline sentiment)
4. Contract count ratios (last resort)

ACTIVITY SCORING ADAPTATION:
- Full data: volume×1000 + OI×10 + premium/100
- Limited data: volume×1000 + fallback calculations
- Always provides meaningful activity ranking
